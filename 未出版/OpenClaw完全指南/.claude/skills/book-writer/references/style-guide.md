# 写作风格指南

> **核心原则**：科技类书籍要通俗易懂、深入浅出、实践+理论并行（实践为主）
> **本书主题**：OpenClaw自进化AI实战指南

---

## 一、整体风格定位

### 1.1 通俗易懂
- **避免学术腔**：不写"本节将探讨""综上所述""研究表明"
- **用大白话讲复杂概念**：先让读者听懂，再谈深度
- **技术术语必须解释**：不要假设读者已经懂
- **多用短句**：一句一个意思，避免长难句

### 1.2 深入浅出
- **从具体现象入手**：先描述问题/场景，再引出概念
- **类比生活化**：用日常经验解释技术原理
- **层层递进**：由浅入深，像剥洋葱一样展开
- **先实践后理论**：让读者先动手，再理解原理

### 1.3 实践为主
- **7:3 实践理论比**：70% 实操步骤 + 30% 原理解释
- **即学即用**：每个知识点配可运行的示例
- **真实场景**：不用"foo/bar"示例，用真实项目案例
- **完整可复现**：代码、命令、配置都能直接复制使用

---

## 二、语言规范

### 2.1 基础规范
- **主语言**：中文
- **技术术语保留英文**：OpenClaw、Gateway、Agent、Skill、MCP、GEPA
- **首次出现**的英文术语必须附中文解释

**正确示例**：
> OpenClaw（开源自进化AI助手平台）是一个让AI从"对话者"变为"执行者"的项目。

**错误示例**：
> OpenClaw 是一个 AI 平台。（读者不知道 OpenClaw 是什么）

### 2.2 人称与语气
- **第二人称"你"为主**："你可以这样做""你会发现"
- **对话式**：像朋友聊天，不要居高临下
- **适度幽默**：可以用轻松的比喻，但不油腻
- **承认局限**："这个方法不是万能的……"

---

## 三、文章结构（每节标准模板）

### 3.1 开场钩子（10% 字数）
**目的**：抓住注意力，让读者觉得"这和我有关"

**常用开场方式**：
1. **痛点开场**："每天处理100封邮件要花 2 小时？"
2. **场景开场**："想象一下，AI自动帮你砍价买车……"
3. **悬念开场**："有个项目72小时获得60,000 stars……"
4. **对比开场**："以前需要一整天，现在只要喊一声"

### 3.2 动手实践（50% 字数）
**目的**：让读者立刻动手，获得成就感

**内容**：
- 完整的操作步骤
- 可复制的代码/命令
- 截图或终端输出示例
- 常见问题和解决

**格式**：
```markdown
## 动手试试

第一步：安装 xxx
```bash
命令行代码
```

第二步：配置 xxx
```yaml
配置文件
```

你会看到如下输出：
```
终端输出示例
```
```

### 3.3 原理解析（30% 字数）
**目的**：解释"为什么这样做"

**时机**：在实践之后，读者有了感性认识再讲原理

**方式**：
- 用类比解释（"这就像……"）
- 图解机制（流程图、架构图）
- 对比不同方案（"这样做比那样做好在哪"）

### 3.4 进阶提示（10% 字数）
**目的**：引导读者继续探索

**内容**：
- 扩展应用场景
- 常见坑和避坑指南
- 预告下一节内容
- 思考题或练习

## 四、表达技巧

### 4.1 类比运用（必须掌握）
把抽象概念具象化：

| 技术概念 | 生活类比 |
|----------|----------|
| OpenClaw | 你的"数字员工团队" |
| Gateway | 公司的"前台接待" |
| Agent | 具体的"员工" |
| GEPA循环 | 员工的"工作-复盘-成长"循环 |
| Skill | 员工的"专业技能" |
| Moltbook | AI的"社交圈" |
| 记忆系统 | 员工的"笔记本" |

### 4.2 场景化表达
让读者身临其境：

**好**：
> 你刚下班，突然想起明天要飞北京。你打开手机给OpenClaw发消息："帮我值机"，然后AI自动帮你完成了一切……

**不好**：
> OpenClaw 支持自动值机功能。

### 4.3 节奏控制
- **段落长度**：3-5 句为宜，不超过 8 句
- **图文比例**：每 500 字至少一个图表/代码块
- **紧张与放松**：复杂概念后接轻松例子
- **信息密度**：不要堆叠术语，一次讲清一个点

## 五、代码规范

### 5.1 代码必须可运行
- 实际测试过，不是凭空写的
- 标注运行环境（Node.js 22+、Python 3.10+ 等）
- 必要的外部依赖要说明安装方式

### 5.2 代码注释
- 用中文注释解释关键逻辑
- 标注可能出错的点
- 解释输出结果

```python
# 正确示例
import requests

# 注意：需要先设置 OPENCLAW_API_KEY 环境变量
api_key = os.environ.get("OPENCLAW_API_KEY")
if not api_key:
    raise ValueError("请设置 OPENCLAW_API_KEY 环境变量")

# 发送请求到本地Gateway
curl -X POST http://localhost:18789/api/message \
  -H "Authorization: Bearer $api_key" \
  -d '{"message": "你好"}'
```

### 5.3 代码展示层次
复杂代码分步骤展示：
1. 先展示完整效果
2. 再拆解关键部分
3. 最后讲扩展用法

## 六、字数与深度

| 内容类型 | 字数 | 深度要求 |
|----------|------|----------|
| 序章/终章 | 2000-3000 | 故事化叙述，引发兴趣 |
| 理论章节（第1-3章） | 2000-4000 | 有原理有案例，逐步深入 |
| 实战章节（第4-8章） | 1500-5000 | 步骤清晰，可立即动手 |
| 进阶章节（第9-11章） | 2500-3500 | 深入话题，有思考价值 |
| 展望章节（第12-13章） | 1500-2500 | 前瞻性思考 |
| 附录 | 1000-1500 | 快速查阅，简洁明了 |

## 八、禁忌清单

### 绝对禁止
- [ ] 不写"本节将介绍……""综上所述……"
- [ ] 不堆砌术语不加解释
- [ ] 不假设读者有前置知识
- [ ] 不用"显然""易得""简单"
- [ ] 不写无法运行的示例代码

### 尽量避免
- [ ] 超过 5 行的纯文字段落
- [ ] 一次引入超过 3 个新概念
- [ ] 只有理论没有实践
- [ ] 只有代码没有解释
- [ ] 使用 foo/bar/baz 作为示例
