# 写作风格指南

> **核心原则**：科技类书籍要通俗易懂、深入浅出、实践+理论并行（实践为主）

---

## 一、整体风格定位

### 1.1 通俗易懂
- **避免学术腔**：不写"本节将探讨""综上所述""研究表明"
- **用大白话讲复杂概念**：先让读者听懂，再谈深度
- **技术术语必须解释**：不要假设读者已经懂
- **多用短句**：一句一个意思，避免长难句

### 1.2 深入浅出
- **从具体现象入手**：先描述问题/场景，再引出概念
- **类比生活化**：用日常经验解释技术原理
- **层层递进**：由浅入深，像剥洋葱一样展开
- **先实践后理论**：让读者先动手，再理解原理

### 1.3 实践为主
- **7:3 实践理论比**：70% 实操步骤 + 30% 原理解释
- **即学即用**：每个知识点配可运行的示例
- **真实场景**：不用"foo/bar"示例，用真实项目案例
- **完整可复现**：代码、命令、配置都能直接复制使用

---

## 二、语言规范

### 2.1 基础规范
- **主语言**：中文
- **技术术语保留英文**：Agent Skill、MCP、SKILL.md、Token、OPC
- **首次出现**的英文术语必须附中文解释

**正确示例**：
> Agent Skill（智能体技能，简称 Skill）是 Claude 的能力封装单元。

**错误示例**：
> Skill 是一种能力封装单元。（读者不知道 Skill 是什么）

### 2.2 人称与语气
- **第二人称"你"为主**："你可以这样做""你会发现"
- **对话式**：像朋友聊天，不要居高临下
- **适度幽默**：可以用轻松的比喻，但不油腻
- **承认局限**："这个方法不是万能的……"

---

## 三、文章结构（每节标准模板）

### 3.1 开场钩子（10% 字数）
**目的**：抓住注意力，让读者觉得"这和我有关"

**常用开场方式**：
1. **痛点开场**："每天写周报要花 2 小时？"
2. **场景开场**："想象一下，周一早上打开 Claude……"
3. **悬念开场**："有个方法可以让效率提升 87.5%……"
4. **对比开场**："以前需要一天，现在只要一小时"

### 3.2 动手实践（50% 字数）
**目的**：让读者立刻动手，获得成就感

**内容**：
- 完整的操作步骤
- 可复制的代码/命令
- 截图或终端输出示例
- 常见问题和解决

**格式**：
```markdown
## 动手试试

第一步：安装 xxx
```bash
命令行代码
```

第二步：配置 xxx
```yaml
配置文件
```

你会看到如下输出：
```
终端输出示例
```
```

### 3.3 原理解析（30% 字数）
**目的**：解释"为什么这样做"

**时机**：在实践之后，读者有了感性认识再讲原理

**方式**：
- 用类比解释（"这就像……"）
- 图解机制（流程图、架构图）
- 对比不同方案（"这样做比那样做好在哪"）

### 3.4 进阶提示（10% 字数）
**目的**：引导读者继续探索

**内容**：
- 扩展应用场景
- 常见坑和避坑指南
- 预告下一节内容
- 思考题或练习

---

## 四、表达技巧

### 4.1 类比运用（必须掌握）
把抽象概念具象化：

| 技术概念 | 生活类比 |
|----------|----------|
| Skill | 数字员工/岗位手册 |
| Progressive Disclosure | 菜单的"更多选项" |
| MCP | 手机的充电接口标准 |
| Subagent | 独立工作室 |
| Context Window | 工作记忆/短期记忆 |

### 4.2 场景化表达
让读者身临其境：

**好**：
> 你刚接手一个老项目，代码库有 10 万行，完全不知道从哪开始。这时候你可以……

**不好**：
> codebase-visualizer 可以生成项目结构图。

### 4.3 节奏控制
- **段落长度**：3-5 句为宜，不超过 8 句
- **图文比例**：每 500 字至少一个图表/代码块
- **紧张与放松**：复杂概念后接轻松例子
- **信息密度**：不要堆叠术语，一次讲清一个点

---

## 五、代码规范

### 5.1 代码必须可运行
- 实际测试过，不是凭空写的
- 标注运行环境（Python 3.10+、Node 18+ 等）
- 必要的外部依赖要说明安装方式

### 5.2 代码注释
- 用中文注释解释关键逻辑
- 标注可能出错的点
- 解释输出结果

```python
# 正确示例
import requests

# 注意：需要先设置 ANTHROPIC_API_KEY 环境变量
api_key = os.environ.get("ANTHROPIC_API_KEY")
if not api_key:
    raise ValueError("请设置 ANTHROPIC_API_KEY 环境变量")

response = requests.post(
    "https://api.anthropic.com/v1/messages",
    headers={"x-api-key": api_key},  # 认证头
    json={"model": "claude-4-opus", "max_tokens": 4096}
)
# 如果返回 401，说明 API Key 无效
```

### 5.3 代码展示层次
复杂代码分步骤展示：
1. 先展示完整效果
2. 再拆解关键部分
3. 最后讲扩展用法

## 六、字数与深度

| 内容类型 | 字数 | 深度要求 |
|----------|------|----------|
| 入门引导（引言、1.1.x） | 1500-2000 | 零基础可读，少理论多故事 |
| 核心概念（1.2.x、2.x） | 2000-3000 | 有原理有实践，逐步深入 |
| 技能介绍（3.x、4.x） | 800-1200 | 快速上手，步骤清晰 |
| 高阶实战（第三章） | 2000-3000 | 完整案例，解决实际问题 |

---

## 七、比喻清单（持续更新）

书中已确立的比喻要延续：
- "游戏存档" → Skill 进化机制
- "数字员工" → Skill 的角色定位
- "指挥官" → OPC 使用 Skill 的状态
- "每周一失忆" → 无记忆 AI 的痛点
- "渐进式披露" → 像餐厅菜单一样分层展示

---

## 八、禁忌清单

### 绝对禁止
- [ ] 不写"本节将介绍……""综上所述……"
- [ ] 不堆砌术语不加解释
- [ ] 不假设读者有前置知识
- [ ] 不用"显然""易得""简单"
- [ ] 不写无法运行的示例代码

### 尽量避免
- [ ] 超过 5 行的纯文字段落
- [ ] 一次引入超过 3 个新概念
- [ ] 只有理论没有实践
- [ ] 只有代码没有解释
- [ ] 使用 foo/bar/baz 作为示例

---

## 九、检查清单（写作后自查）

- [ ] 开头能否在 10 秒内抓住读者？
- [ ] 是否提供了可运行的代码/命令？
- [ ] 技术术语首次出现是否有解释？
- [ ] 是否有生活化类比帮助理解？
- [ ] 段落是否过长（超过 8 句）？
- [ ] 是否有足够的图表/代码块？
- [ ] 结尾是否引导读者继续阅读？
- [ ] 整体是否 7 分实践 3 分理论？
