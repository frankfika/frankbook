# 2.2.2 核心字段：name和description

## name字段：Skill的身份标识

name是Skill的唯一标识符，它在整个Skill生态中扮演着身份证的角色。

### 命名规范

name字段必须遵循以下规范：

**格式要求**：
- 只能使用小写字母、数字和连字符（`-`）
- 必须以字母开头
- 不能以连字符开头或结尾
- 最大长度：64个字符

**有效示例**：
```yaml
name: code-reviewer
name: python-linter
name: doc-generator-v2
name: git-commit-helper
```

**无效示例**：
```yaml
name: Code Reviewer        # 错误：包含大写字母和空格
name: 代码审查              # 错误：包含非ASCII字符
name: -code-reviewer       # 错误：以连字符开头
name: code_reviewer        # 错误：使用下划线而非连字符
name: a                    # 警告：过于简短，不具描述性
```

### 命名的艺术

一个好的name应该：

**1. 具有描述性**

```yaml
# 好：一看就知道是做什么的
name: security-scanner
name: performance-profiler

# 差：过于笼统
name: tool
name: helper
```

**2. 使用领域术语**

```yaml
# 好：使用行业标准术语
name: unit-test-generator
name: api-doc-builder

# 差：生造词汇
name: test-maker
name: doc-creator
```

**3. 保持一致性**

如果你有一组相关的Skills，使用一致的命名风格：

```yaml
# 一致的命名
name: review-code-python
name: review-code-javascript
name: review-code-go

# 而不是
name: python-code-reviewer
name: js-code-checker
name: golang-reviewer
```

**4. 避免命名冲突**

在命名前，建议检查是否已有相同或相似的Skill：

```bash
# 列出已安装的Skills
claude skills list

# 搜索特定名称
claude skills search code-review
```

如果名称冲突，可以添加前缀或后缀区分：

```yaml
name: acme-code-reviewer      # 公司名称作为前缀
name: code-reviewer-advanced  # 添加描述性后缀
```

### name的使用场景

name字段在多个场景中被使用：

**1. 斜杠命令**

当`user-invocable: true`时，name成为斜杠命令：

```yaml
name: code-reviewer
# 用户可以通过输入 /code-reviewer 调用
```

**2. 程序调用**

在脚本或其他Skill中引用：

```bash
# 使用特定Skill处理文件
claude skill run code-reviewer --file app.py
```

**3. 依赖声明**

一个Skill可以声明依赖其他Skill：

```yaml
name: full-stack-reviewer
dependencies:
  - frontend-reviewer
  - backend-reviewer
```

## description字段：触发条件的核心
description是YAML Frontmatter中最重要的字段之一，它直接决定了Skill何时被激活。

### description的作用

当用户输入一个请求时，Claude会：

1. 分析用户请求的语义
2. 读取所有已安装Skill的description
3. 计算用户请求与每个description的匹配度
4. 如果某个Skill的匹配度超过阈值，激活该Skill

因此，description本质上是一个**触发条件声明**。

### 编写好的description

**原则1：使用第三人称**

description描述的是"这个Skill应该在什么情况下被使用"，因此使用第三人称：

```yaml
# 好：描述何时使用
# "当用户需要审查代码时"
description: When user asks to review code for quality and security issues

# 差：描述Skill能做什么
# "这个Skill可以审查代码"
description: This skill reviews code quality
```

**原则2：包含关键词**

在description中包含用户可能使用的关键词：

```yaml
# 好：覆盖了多种表达方式
description: When user asks to review, check, or audit code for bugs, security vulnerabilities, or style issues

# 差：过于简单，可能遗漏触发
description: When user wants code review
```

**原则3：具体而非抽象**

```yaml
# 好：具体说明触发场景
description: When user asks to generate a weekly report summarizing git commits, completed tasks, and key metrics

# 差：过于抽象
description: When user needs report generation
```

**原则4：覆盖边界情况**

考虑用户可能的不同表达方式：

```yaml
description: |
  When user asks to:
  - Review code or conduct code review
  - Check code quality or analyze code
  - Find bugs or security issues in code
  - Improve code structure or refactoring suggestions
```

### description的长度限制
description字段有最大长度限制：**1024个字符**。

这个限制是为了：
- 保证匹配计算的性能
- 鼓励编写简洁明了的描述
- 避免过长的description影响启动速度

如果1024个字符不够用，可以：
1. 精简描述，保留最核心的触发条件
2. 将详细的说明移到SKILL.md正文
3. 拆分成多个更具体的Skills

### 正反示例对比

**示例1：代码审查Skill**

```yaml
# 差的description
description: Code reviewer skill

# 问题：
# 1. 没有说明什么时候使用
# 2. 使用了第一人称"skill"
# 3. 没有关键词

# 好的description
description: When user asks to review Python or JavaScript code for quality, security vulnerabilities, performance issues, and adherence to best practices

# 优点：
# 1. 使用第三人称
# 2. 包含语言信息（Python/JavaScript）
# 3. 列出检查的具体方面
```

**示例2：文档生成Skill**

```yaml
# 差的description
description: Generate documentation for projects

# 问题：
# 1. 使用祈使句而非第三人称
# 2. "projects"过于宽泛

# 好的description
description: When user needs to generate API documentation from code, create README files, or write technical documentation for software projects

# 优点：
# 1. 使用第三人称
# 2. 具体说明文档类型
# 3. 涵盖多个触发场景
```

## 其他元数据字段

除了name和description，YAML Frontmatter还支持其他字段。

### type字段

指定Skill的类型：

```yaml
type: skill    # 普通Skill（默认）
type: system   # 系统Skill（保留）
type: template # 模板Skill（可继承）
```

大多数Skill使用默认的`skill`类型。

### disabled字段

控制Skill是否启用：

```yaml
disabled: false  # 启用（默认）
disabled: true   # 禁用
```

禁用的Skill不会被加载，也不会参与触发匹配。

### user-invocable字段

控制用户是否可以通过斜杠命令手动调用：

```yaml
user-invocable: true   # 可以通过 /skill-name 调用
user-invocable: false  # 只能通过AI自动触发（默认）
```

对于高频使用的Skill，建议设置为true；
对于后台辅助的Skill，建议保持false。

### disable-model-invocation字段

控制是否禁止AI自动触发：

```yaml
disable-model-invocation: false  # AI可以自动触发（默认）
disable-model-invocation: true   # 必须用户手动触发
```

对于高风险操作（如删除数据、部署应用），建议设置为true，确保用户明确同意。

### allowed-tools字段

指定该Skill可以使用的工具：

```yaml
allowed-tools:
  - Read
  - Write
  - Bash
  - Grep
```

这是一个安全机制，限制了Skill的能力范围。即使Skill的指令要求使用某个工具，如果不在allowed-tools列表中，也会失败。

### context字段

指定Skill的执行上下文：

```yaml
context: inline  # 在主对话上下文中执行（默认）
context: fork    # 在独立的fork上下文中执行
```

`fork`提供更好的隔离性，适合复杂或长时间的Skill；
`inline`提供更好的连续性，适合简单快捷的Skill。

### model字段

指定执行该Skill使用的模型：

```yaml
model: claude-4-opus      # 使用Opus模型（最强推理）
model: claude-4-sonnet    # 使用Sonnet模型（平衡）
model: claude-4-haiku     # 使用Haiku模型（最快）
```

如果 Skill 需要复杂的推理，使用Opus；
如果 Skill 只是简单的格式转换，使用Haiku可以节省成本。

## 实际案例分析

让我们分析几个实际Skill的元数据配置。

**案例1：Git Commit Skill**

```yaml
---
name: git-commit
description: When user wants to commit changes with a descriptive message following conventional commits format
type: skill
user-invocable: true
disable-model-invocation: false
allowed-tools:
  - Bash
  - Read
  - Grep
context: inline
---
```

分析：
- name简洁明了
- description包含关键词"commit"、"conventional commits"
- user-invocable=true：高频操作，需要斜杠命令
- allowed-tools限制在Git相关操作
- context=inline：简单快速的操作

**案例2：Database Migration Skill**

```yaml
---
name: db-migration
description: When user needs to create, modify, or manage database schema migrations
type: skill
user-invocable: true
disable-model-invocation: true
allowed-tools:
  - Read
  - Write
  - Bash
  - mcp-postgres/query
context: fork
model: claude-4-opus
---
```

分析：
- name使用缩写但清晰
- description覆盖多种迁移操作
- disable-model-invocation=true：数据库操作风险高，需要用户确认
- allowed-tools包含MCP工具，用于实际数据库操作
- context=fork：复杂操作，需要隔离
- model=opus：需要强推理能力处理复杂的迁移逻辑

## 小结

name和description是SKILL.md中最重要的两个字段：

- **name**：Skill的唯一标识符，也是斜杠命令的名称
  - 遵循命名规范（小写、连字符、64字符限制）
  - 具有描述性，使用领域术语
  - 保持命名一致性

- **description**：触发条件的核心声明
  - 使用第三人称
  - 包含关键词
  - 具体而非抽象
  - 1024字符限制

其他字段（type、disabled、user-invocable等）提供了更精细的控制，根据Skill的具体需求合理配置。
