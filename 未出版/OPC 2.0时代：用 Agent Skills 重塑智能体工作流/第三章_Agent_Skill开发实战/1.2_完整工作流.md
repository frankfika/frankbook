# 1.2 完整工作流：从搜索GitHub到Skill固化

## 概述：Skill开发的完整周期

开发一个高质量的Skill并非一蹴而就，而是一个从需求分析、方案调研、原型开发到持续优化的完整周期。本章将详细介绍这个工作流，帮助你建立系统化的Skill开发方法论。

## 阶段一：需求分析与问题定义

### 明确目标

在开始任何开发之前，首先要明确：

**你要解决什么问题？**
- 是重复性的手动操作？
- 是复杂的决策过程？
- 是信息整合和分析？
- 是创意和内容生成？

**谁是你的用户？**
- 是你自己（个人工作流）？
- 是你的团队成员？
- 是更广泛的社区用户？

**成功的标准是什么？**
- 节省时间（节省多少？）
- 提高质量（如何衡量？）
- 降低错误率（减少多少？）

### 场景拆解

将复杂需求拆解为可执行的具体场景：

例如，"帮我处理文档"这个需求过于模糊，应该拆解为：
- 从PDF提取表格数据
- 将Word转换为Markdown
- 批量添加水印
- 合并多个PDF文件

每个具体场景都应该有明确的输入、输出和成功标准。

## 阶段二：GitHub搜索与方案调研

### 系统化搜索策略

**关键词矩阵**。构建多组关键词组合，全面搜索可能的解决方案：

| 核心词 | 技术词 | 语言词 | 组合 |
|--------|--------|--------|------|
| pdf | extraction | python | "python pdf extraction" |
| document | converter | javascript | "javascript document converter" |
| image | batch processing | go | "go image batch processing" |

**深度挖掘**。找到一个相关项目后：
1. 查看其依赖的项目
2. 查看引用它的项目
3. 查看同一作者的其他项目
4. 查看相关的Awesome列表

### 方案对比与选择

创建对比表格，客观评估每个候选方案：

| 项目 | Stars | 活跃度 | 功能覆盖 | 易用性 | 许可证 |
|------|-------|--------|----------|--------|--------|
| 方案A | 5000 | 高 | 80% | 易 | MIT |
| 方案B | 1200 | 中 | 95% | 中 | Apache |
| 方案C | 300 | 低 | 60% | 易 | GPL |

选择时要考虑：
- **功能匹配度**。是否覆盖核心需求？
- **长期可维护性**。项目是否活跃？
- **集成难度**。API是否清晰？文档是否完善？
- **性能要求**。是否能满足处理速度需求？

## 阶段三：原型开发与快速验证

### 最小可行Skill（MVS）

不要一开始就追求完美，先构建一个能工作的最小版本：

1. **核心功能优先**。只实现最关键的一个功能。
2. **硬编码配置**。先不要考虑灵活的配置。
3. **简单错误处理**。先假设输入是正确的。
4. **快速迭代**。能运行后立即测试。

### 验证清单

原型完成后，验证以下要点：

- [ ] 核心功能是否正常工作？
- [ ] 处理速度是否可接受？
- [ ] 输出质量是否达到预期？
- [ ] 是否容易使用（自然语言指令）？
- [ ] 错误信息是否清晰？

如果验证失败，回到阶段二重新调研；如果验证通过，进入下一阶段。

## 阶段四：Skill结构设计与文档编写

### YAML Frontmatter设计

精心设计Skill的元数据：

```yaml
---
name: pdf-table-extractor
description: Extract tables from PDF files and convert to CSV/Excel
user-invocable: true
allowed-tools:
  - Read
  - Bash
  - Write
model: haiku  # 对于简单任务使用轻量级模型
---
```

关键考虑：
- **name**。简洁、唯一、易记。
- **description**。清晰描述触发条件和功能。
- **allowed-tools**。最小权限原则，只授予必要的工具。

### Markdown正文编写

按照三层结构设计正文：

**第一层：触发条件**
- 列出明确的触发短语
- 描述适用场景
- 说明不适用的情况

**第二层：执行步骤**
- 详细但清晰的步骤说明
- 代码示例
- 注意事项

**第三层：参考资料**
- 技术文档链接
- 常见问题和解决方案
- 进阶用法

### 脚本和引用文件

创建scripts目录存放可执行脚本：

```
skills/
└── pdf-table-extractor/
    ├── SKILL.md
    ├── scripts/
    │   ├── extract_tables.py
    │   └── convert_format.py
    └── references/
        ├── supported_formats.md
        └── troubleshooting.md
```

## 阶段五：测试与调试

### 测试用例设计

设计覆盖各种场景的测试用例：

**正常场景**：
- 标准PDF文件
- 常见表格格式
- 典型文件大小

**边界场景**：
- 空文件
- 超大文件
- 扫描版PDF（无文本层）
- 复杂布局的表格

**异常场景**：
- 损坏的文件
- 不支持的格式
- 权限不足的文件

### 调试技巧

当Skill不工作时：

1. **检查YAML语法**。使用在线YAML验证器。
2. **验证工具权限**。确认allowed-tools包含所需工具。
3. **查看错误日志**。Bash命令的输出往往包含有用信息。
4. **简化测试**。先用最简单的输入测试核心功能。
5. **逐步增加复杂度**。确保每个新增功能都正常工作。

## 阶段六：固化与版本管理

### Git版本控制

使用Git管理Skill的演进：

```bash
# 初始化仓库
git init

# 首次提交
git add .
git commit -m "Initial version: basic PDF table extraction"

# 功能迭代
git add .
git commit -m "Add support for multi-page tables"
```

### 版本标签

为重要的里程碑打标签：

```bash
# v1.0：基础功能完成
git tag -a v1.0 -m "Basic functionality complete"

# v1.1：增加新功能
git tag -a v1.1 -m "Add Excel output format"
```

### 文档更新

每次迭代都要同步更新文档：
- 更新CHANGELOG
- 更新使用示例
- 更新已知问题

## 阶段七：部署与分享

### 本地部署

将Skill安装到Claude Code：

```bash
# 使用skill命令安装
claude skill install ./pdf-table-extractor

# 或手动复制到skills目录
cp -r pdf-table-extractor ~/.claude/skills/
```

### 团队分享

在团队中共享Skill：

1. **Git仓库**。将Skill推送到团队Git仓库。
2. **安装说明**。提供清晰的安装步骤。
3. **使用培训**。演示核心功能和最佳实践。
4. **反馈收集**。建立反馈渠道，持续改进。

### 开源发布

如果Skill具有通用价值，考虑开源：

1. **选择合适的LICENSE**（推荐MIT或Apache 2.0）。
2. **编写README**。包含安装、使用、贡献指南。
3. **创建示例**。提供演示视频或GIF。
4. **推广分享**。在技术社区分享你的作品。

## 工作流优化：建立个人Skill库

### 分类体系

建立清晰的Skill分类：

```
skills/
├── 01-document/       # 文档处理
├── 02-code/           # 代码开发
├── 03-data/           # 数据分析
├── 04-content/        # 内容创作
├── 05-communication/  # 沟通协作
└── 99-personal/       # 个人专属
```

### 快速检索

建立Skill索引文档：

```markdown
# Skill索引

## 文档处理
- [[pdf-extractor]] - PDF内容提取
- [[doc-converter]] - 格式转换
- [[report-generator]] - 报告生成

## 代码开发
- [[code-reviewer]] - 代码审查
- [[test-generator]] - 测试生成
- [[refactor-helper]] - 重构辅助
```

### 持续优化

定期回顾和优化你的Skill库：
- 删除不再使用的Skill
- 合并功能重复的Skill
- 更新过时的依赖
- 改进使用体验

## 小结

完整的Skill开发工作流包括：

1. **需求分析**。明确问题、用户和成功标准。
2. **方案调研**。系统搜索GitHub，选择最佳方案。
3. **原型开发**。构建MVS，快速验证可行性。
4. **结构设计**。编写清晰的Skill文档和脚本。
5. **测试调试**。覆盖各种场景，确保稳定可靠。
6. **版本管理**。使用Git管理演进过程。
7. **部署分享**。在团队或社区中推广使用。

遵循这个工作流，你可以系统性地开发高质量的Skill，避免重复造轮子，持续提升个人和团队的生产力。
